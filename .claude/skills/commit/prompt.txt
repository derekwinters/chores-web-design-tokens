When invoked with /commit, follow this flow:

## Step 1: Run Tests
- Run: npm test (runs the Style Dictionary build via node build.js, then vitest run)
- If any fail, stop and report failures in detail
- Do NOT proceed if tests fail

## Step 2: Review Changes
- Run: git diff --staged (to see staged changes)
- Run: git status (to see working tree)
- Understand what changed and why

## Step 3: Determine Type and Scope
From the actual changes, derive:
- Type: feat | fix | refactor | test | docs | chore | style | perf | ci
- Scope: tokens | themes | build | android | test (pick most relevant)

Token-change severity → commit type (same table as CLAUDE.md `## Releases`
and README.md — reference it, don't fork it):
- Token added → feat: (MINOR)
- Token value changed → fix: (PATCH)
- Token removed or renamed → feat!: or a BREAKING CHANGE: footer (MAJOR)
  Still required pre-1.0 despite bump-minor-pre-major, so history stays
  correct once the package reaches 1.0.0.

Examples:
- Adding a semantic token → feat(tokens)
- Changing a token value → fix(tokens)
- Renaming/removing a token → feat!(tokens) or a BREAKING CHANGE: footer
- Adding a runtime theme palette → feat(themes)
- Updating the Style Dictionary config → refactor(build) or chore(build)
- Updating tests → test
- Updating docs → docs

## Step 4: Write Commit
- Subject: <type>(<scope>): description
- Rules:
  - ≤72 characters
  - lowercase
  - no period at end
  - imperative mood ("add" not "added")
- Body: only if "why" is non-obvious
- Include Co-Authored-By footer

## Step 5: Execute
- Create and execute the commit
- Verify with git log

## Important
- MUST have all tests passing before commit
- MUST use imperative mood
- MUST match Conventional Commits format
